# Claude Code Markdown フォーマット仕様

更新日: 2026-03-06

本仕様は [GFM (GitHub Flavored Markdown) Spec](https://github.github.com/gfm/) をベースに、Claude Code が Markdown ファイルを出力する際に使用する構文を定義したものである。GFM は [CommonMark](https://spec.commonmark.org/) の厳密なスーパーセットであり、本仕様はその中から必要な構文のみを抽出・限定している。

---

## 1. ATX 見出し (Section 4.2, Examples 32-49)

```markdown
# 見出し1
## 見出し2
### 見出し3
```

- `#` を1～6個使用（7個以上は見出しではない）
- `#` の直後に空白または行末が必須（`#5` は見出しではない）
- 末尾の `#` は任意。前にスペースが必須

## 2. 水平線 (Section 4.1, Examples 13-31)

```markdown
---
```

- 3個の `-` で記述する
- 前後に空行が必要

## 3. フェンスコードブロック (Section 4.5, Examples 89-117)

> コードフェンスとは、3個以上の連続するバッククォート文字（`` ` ``）の並びである。

- 開始フェンス: 3個以上の `` ` ``
- 終了フェンス: 開始以上の個数の `` ` ``
- インフォ文字列（言語指定）は開始フェンスのみ
- インフォ文字列にバッククォート不可
- 開始フェンスが N スペースインデントの場合、内容行から最大 N スペースを除去
- 閉じられなかったフェンスはドキュメント末尾で閉じる

## 4. テーブル (GFM 拡張, Section 4.10, Examples 198-205)

### 4.1 基本構造

```markdown
| ヘッダ1 | ヘッダ2 | ヘッダ3 |
| --- | --- | --- |
| データ1 | データ2 | データ3 |
```

ヘッダ行 + セパレータ行 + 0個以上のデータ行で構成。

### 4.2 セパレータ行（delimiter row）

> セパレータ行は、ハイフン（`-`）のみで構成されるセルからなり、オプションで先頭・末尾にコロン（`:`）を付与して左寄せ・右寄せ・中央寄せを指定できる。

| 構文 | 配置 |
| --- | --- |
| `---` | デフォルト（左） |
| `:---` | 左寄せ |
| `---:` | 右寄せ |
| `:---:` | 中央寄せ |

### 4.3 セル内のルール

- セル内容の前後の空白はトリムされる
- セル内でパイプを使う場合は `\|` でエスケープ
- セル内ではインライン要素が解析される（コードスパン、強調等）
- ブロック要素は使用不可

### 4.4 セル数の不一致

- ヘッダ行とセパレータ行のセル数が不一致 → テーブルと認識されない (Example 203)
- データ行がヘッダより少ない → 空セルで補充
- データ行がヘッダより多い → 超過分は無視

### 4.5 具体例

**Example 200**: セル内のパイプエスケープ

```markdown
| f\|oo  |
| ------ |
| b `\|` az |
| b **\|** im |
```

## 5. コードスパン (Section 6.3, Examples 328-349)

> バッククォート文字列とは、1個以上の連続するバッククォート文字（`` ` ``）で、前後にバッククォートがないものである。コードスパンはバッククォート文字列で始まり、同じ長さのバッククォート文字列で終わる。

### 5.1 基本ルール

- 開始と終了で**同じ数**のバッククォートを使用
- 開始デリミタと同じ数のバッククォート文字列が最初に見つかった位置で閉じる
- バックスラッシュエスケープは効かない（内容はリテラル）
- 改行はスペースに変換される

基本的なコードスパン:

```markdown
`code` → code
```

バックスラッシュはリテラル扱い:

```markdown
`foo\bar` → foo\bar（エスケープされない）
```

改行はスペースに変換:

```markdown
`foo
bar` → foo bar
```

### 5.2 先頭・末尾スペースの扱い

> 結果の文字列が先頭と末尾の両方がスペースで、かつスペースのみで構成されていない場合、先頭と末尾から各1文字のスペースを除去する。

- 先頭と末尾の**両方**がスペースで、かつスペースのみではない → 各1文字除去
- 片方だけスペース → 除去しない
- スペースのみ → 除去しない

```markdown
`` `foo` ``   → `foo`（前後スペース除去で ` が内容に含まれる）
`` ` ``       → `（前後スペース除去）
`  ``  `      →  ``（前後各1スペース除去、内側スペースは残る）
` `           → ` `（スペースのみなので除去しない）
```

### 5.3 バッククォート数の決定方法

内容に含まれるバッククォートの連続と衝突しない最小数を選ぶ。

| 内容 | 必要なデリミタ数 | 記法 |
| --- | --- | --- |
| バッククォートなし | 1個 | `` `code` `` |
| 単独の `` ` `` あり | 2個 | ``` `` foo`bar `` ``` |
| `` `` `` あり、単独 `` ` `` なし | 1個 | `` ` foo``bar ` `` |

```markdown
`code`              ← バッククォートなし → 1個で囲む
`` foo`bar ``       ← 内容に1個の ` → 2個で囲む
` foo``bar `        ← 内容に2個の `` だが1個の ` なし → 1個で囲む
```

## 6. リスト (Section 5.2-5.4)

**箇条書き**: `-`
**順序付き**: `1.`（最大9桁）

密なリスト（項目間に空行なし）:

```markdown
- 項目1
- 項目2
- 項目3
```

緩いリスト（項目間に空行あり、各項目が `<p>` で囲まれる）:

```markdown
- 項目1

- 項目2

- 項目3
```

順序付きリスト:

```markdown
1. 項目1
2. 項目2
3. 項目3
```

ネストしたリスト（4スペースインデント）:

```markdown
- 項目1
    - サブ項目1
    - サブ項目2
- 項目2
```

## 7. タスクリスト (GFM 拡張, Section 5.3)

```markdown
- [ ] 未完了タスク
- [x] 完了タスク
```

- リスト項目の先頭に `[ ]`（未完了）または `[x]`（完了）を付与
- `[` と `]` の間はスペースまたは `x` のみ
- 箇条書き・順序付きリストのどちらでも使用可能
- ネスト対応

ネストしたタスクリスト:

```markdown
- [ ] 親タスク
    - [x] サブタスク1
    - [ ] サブタスク2
```

## 8. ブロック引用 (Section 5.1)

```markdown
> 引用テキスト
> 続きの行
```

- `>` で開始
- ネストが可能

```markdown
> 外側の引用
> > ネストした引用
```

- 段落の継続行では `>` を省略可能（遅延継続）

```markdown
> 引用の開始
続きの行（> を省略）
```

## 9. 強調・太字 (Section 6.4)

```markdown
*強調*
**太字**
***太字強調***
```

## 10. 取り消し線 (GFM 拡張, Section 6.5, Example 490)

```markdown
~~取り消し~~
```

- `~~` で囲む
- 内容は空であってはならない

## 11. リンク・画像 (Section 6.6-6.7)

```markdown
[テキスト](URL "タイトル")
![代替テキスト](URL "タイトル")
```

## 12. バックスラッシュエスケープ (Section 6.1, Examples 296-305)

> すべての ASCII 句読文字はバックスラッシュでエスケープできる。

```
! " # $ % & ' ( ) * + , - . / : ; < = > ? @ [ \ ] ^ _ ` { | } ~
```

| コンテキスト | エスケープ |
| --- | --- |
| 通常テキスト | 有効 |
| コードスパン内 | **無効**（リテラル扱い） |
| コードブロック内 | **無効**（リテラル扱い） |

## 13. エンティティ参照 (Section 6.2, Examples 306-327)

> エンティティ参照および数値文字参照は、コードブロックおよびコードスパン内では認識されない。

| 種類 | 構文 | 例 |
| --- | --- | --- |
| HTML エンティティ | `&name;` | `&amp;` → `&`, `&lt;` → `<`, `&gt;` → `>` |
| 10進数値参照 | `&#decimal;` | `&#35;` → `#` |
| 16進数値参照 | `&#xhex;` | `&#x23;` → `#` |

| コンテキスト | 展開 |
| --- | --- |
| 通常テキスト | される |
| コードスパン内 | **されない** |
| コードブロック内 | **されない** |

## 14. 改行 (Section 6.12-6.13)

| 種類 | 構文 | 出力 |
| --- | --- | --- |
| ハードブレーク | `\` | `<br />` |
| ソフトブレーク | 通常の改行 | スペースに変換 |

ハードブレーク:

```markdown
行1\
行2
```

→ 行1`<br />`行2（同上）

ソフトブレーク:

```markdown
行1
行2
```

→ 行1 行2（スペースで連結される）

段落区切り（空行）:

```markdown
段落1

段落2
```

→ 別々の `<p>` 要素になる

## 15. マークダウン設計書: 可読性向上ガイドライン

設計書の可読性とメンテナンス性を一貫させるため、以下のルールを適用する。

### 15.1 スペースと改行のルール（視認性）

- **見出しの余白**: 見出し（`##` 等）の上には2行、下には1行の空行を入れる。セクションの区切りを視覚的に明確にするため。
- **句読点での改行**: 原則として「。」の後は改行（ハードブレーク）を入れる。1行を短く保つことで、スマホやブラウザの分割画面での閲覧性を高める。
- **リストの前後**: 箇条書き（リスト）の前後には必ず1行の空行を挟み、前後の本文と癒着させない。

### 15.2 構文と記法（情報の構造化）

- **固有名称のハイライト**: 変数名、ファイル名、ボタン名、テーブル名などのシステム固有名称は、必ずインラインコード（バッククォート）で囲む。
- **補足情報の分離**: 本筋から外れる注釈や補足、注意喚起は、引用ブロック（`>`）を使用して本文と差別化する。
- **図解の優先**: 文字だけで説明が複雑になるロジック（分岐や順序）は、文章で粘らず Mermaid 図を用いて図解する。

### 15.3 リストと階層（論理構造）

- **階層の制限**: 箇条書き（リスト）の深さは2階層までとする。それ以上深くなる場合は、別の子見出し（`###`）を立てて構造を再設計する。
- **インデントの統一**: リスト内での補足やネストは、半角スペース2つ（または4つ）を用いて正確に階層を揃える。
