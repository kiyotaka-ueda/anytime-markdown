# Anytime Markdown ウェブサイト改善計画


更新日: 2026-03-07


## 仮定一覧

本計画は以下の仮定に基づく。\
実際の運用条件と異なる場合は、対応する改善案の優先度・内容を見直すこと。

| ID | 仮定内容 | 影響度 |
| --- | --- | --- |
| H-1 | 本番環境は HTTPS で運用される | 高 |
| H-2 | 月間アクティブユーザーは1,000人未満（小〜中規模） | 中 |
| H-3 | 管理者は1〜3名程度で、CMS 操作頻度は週数回 | 中 |
| H-4 | S3 バケットは非公開で、API 経由でのみアクセス可能 | 高 |
| H-5 | ダークモード使用率は全ユーザーの30〜40% | 中 |
| H-6 | モバイルアクセス比率は40〜50% | 高 |
| H-7 | MUI Dialog はデフォルトでフォーカストラップ・ESC 閉じに対応済み | 低 |


## 現状サマリー

### サイト構成

| ページ | パス | 用途 |
| --- | --- | --- |
| ランディング | `/` | プロダクト紹介、CTA |
| フィーチャー | `/features` | 機能詳細（Markdown 解析表示） |
| ドキュメント一覧 | `/docs` | カード形式の公開ドキュメント一覧 |
| ドキュメント閲覧 | `/docs/view` | Markdown ビューア |
| レイアウト編集 | `/docs/edit` | 管理者用 CMS（D&D カード配置） |
| エディタ | `/markdown` | WYSIWYG Markdown エディタ |
| プライバシー | `/privacy` | プライバシーポリシー |

### 技術スタック

- Next.js 15 App Router / React 19 / TypeScript
- MUI 7 / next-intl / @dnd-kit
- AWS SDK v3（S3）/ Basic Auth
- Tiptap ベースの `editor-core`（モノレポ共有）


### 分析体制

3名のエージェントが独立して現状分析を実施し、課題を突き合わせた。

- **Designer（40件）**: ユーザーフロー、情報設計、インタラクション、レスポンシブ、ビジュアル一貫性
- **A11y（15件）**: WCAG 2.2 AA 基準に基づくアクセシビリティ監査
- **Engineer（25件）**: Next.js ベストプラクティス、セキュリティ、パフォーマンス、保守性

## 改善案


### C-01: D&D のキーボード代替・操作ガイド欠落

**担当観点**: A11y（主）、Designer（副）\
**工数見積**: S\
**期待効果**: キーボードユーザーがカード並べ替え可能になる。\
WCAG 2.1.1（キーボード）、2.5.7（ドラッグ操作）準拠。

**改善内容**:

- `@dnd-kit/sortable` の `KeyboardSensor` + `sortableKeyboardCoordinates` を追加
- ドラッグハンドルに `aria-roledescription="sortable"` と操作説明の `aria-label` を付与
- カード配置エリアのヘッダーに操作説明テキストを追加

> **【A11y 必須】** `KeyboardSensor` の追加は WCAG 2.1.1 準拠に必須。
>
> **【Designer 推奨】** ツールチップでの操作ガイドはオプションだが UX 向上に有効。


### C-02: 入力バリデーション不足

**担当観点**: Engineer（主）\
**工数見積**: M\
**期待効果**: DoS 攻撃・不正データ注入リスクの軽減。

**改善内容**:

- アップロード API にファイルサイズ上限（例: 5MB）を追加
- レイアウト API に JSON スキーマ検証を追加（`zod` 導入）
- タグの文字種・長さ制限（英数字・日本語、各タグ最大30文字、最大10個）
- `src/types/layout.ts` にスキーマ定義を一元化（C-11 と統合）

> **【A11y 推奨】** バリデーションエラー時のメッセージは `aria-live="polite"` で通知すること。


### C-03: XSS リスク（`dangerouslySetInnerHTML`）

**担当観点**: Engineer（主）\
**工数見積**: S\
**期待効果**: フィーチャーページ経由の XSS 攻撃を防止。

**改善内容**:

- DOMPurify の `ALLOWED_ATTR` から `onerror`, `onload` 等のイベントハンドラ属性を明示的に除外
- `FORBID_ATTR: ['onerror', 'onload', 'onclick', 'onmouseover']` を追加
- CSP ヘッダーで `script-src 'strict-dynamic'` を検討


### C-04: 色コントラスト比不足

**担当観点**: A11y（主）、Designer（副）\
**工数見積**: S\
**期待効果**: WCAG 1.4.3（コントラスト比 4.5:1）準拠。

**改善内容**:

- `#e8a012` 背景 + `#1a1a1a` テキスト → コントラスト比約 4.3:1（不足）
- ボタンテキストを `#000000` に変更 → コントラスト比 5.7:1（AA 達成）
- MUI テーマの `contrastText` を明示的に `#000000` に設定
- ダークモードでの半透明背景（`rgba(255,255,255,0.03)`）上のテキストコントラストを検証


### C-05: フィーチャーアイコンの代替テキスト欠落

**担当観点**: A11y（主）\
**工数見積**: XS\
**期待効果**: スクリーンリーダーでフィーチャーアイコンの意味が伝わる。

**改善内容**:

- 各 Icon に `aria-hidden="true"` を設定（テキストラベルが隣接するため装飾アイコン扱い）
- アイコン単独で使用される箇所には `aria-label` を付与


### C-06: Basic 認証の脆弱性

**担当観点**: Engineer（主）\
**工数見積**: L\
**期待効果**: 管理操作のセキュリティ強化。

**改善内容（段階的）**:

- **Phase 1（Quick Win）**: CSRF トークン検証の追加、レート制限（1分あたり10リクエスト）
- **Phase 2（短期）**: Cookie ベースのセッション認証への移行を検討
- **Phase 3（中期）**: NextAuth.js 等の認証ライブラリ導入を検討

> **【Engineer 注記】** H-1（HTTPS 前提）が成立しない場合、Basic 認証は即座に廃止すべき。
>
> **【Designer 質問】** 認証 UI（ログインフォーム）の追加は管理者 UX に影響するが、セキュリティ優先で進めてよいか？
>
> → **合意**: セキュリティ優先。管理者数が少ないため（H-3）、ログインフォームの UX コストは許容範囲。


### C-07: 色のみによる状態伝達

**担当観点**: A11y（主）、Designer（副）\
**工数見積**: S\
**期待効果**: 色覚多様性のあるユーザーもフィルタ状態を把握可能。

**改善内容**:

- 選択中のタグ Chip にチェックマークアイコン（`CheckIcon`）を追加
- `aria-pressed` 属性でトグル状態を明示
- 削除ボタンのホバー時に色変更だけでなくアンダーラインも追加


### C-08: ヒーロー CTA のモバイル配置崩れ

**担当観点**: Designer（主）\
**工数見積**: XS\
**期待効果**: モバイルでの CTA タップ率向上。

**改善内容**:

- CTA ボタン群を `flexWrap: 'wrap'` + `justifyContent: 'center'` に変更
- モバイル時に `width: '100%'` でボタンを全幅化
- 副ボタン（GitHub）は `mt: 1.5` で下に配置


### C-09: フィーチャーページの目次がモバイル非表示

**担当観点**: Designer（主）\
**工数見積**: M\
**期待効果**: モバイルでもドキュメント内ナビゲーションが可能。

**改善内容**:

- モバイル用にフローティング FAB（目次アイコン）を画面右下に配置
- タップで Drawer として目次を表示
- 見出しクリックで該当箇所にスクロール + Drawer 自動閉じ


### C-10: RSC 活用不足

**担当観点**: Engineer（主）\
**工数見積**: M\
**期待効果**: 初回ロードの JS バンドルサイズ削減、LCP 改善。

**改善内容**:

- `SitesBody.tsx`: データ取得をサーバーコンポーネント化し、カード表示のみクライアントに分離
- `LandingBody.tsx`: 静的コンテンツはサーバーレンダリング、テーマ依存部分のみクライアント化

> **【A11y 推奨】** サーバーレンダリング化により初回 HTML にコンテンツが含まれ、スクリーンリーダーの読み上げ開始が早くなる。


### C-11: 型定義の重複・API レスポンス型の未検証

**担当観点**: Engineer（主）\
**工数見積**: S\
**期待効果**: 型不整合によるランタイムエラーの防止、保守性向上。

**改善内容**:

- `src/types/layout.ts` を新規作成し、`LayoutCard`, `LayoutData`, `DocFile` を一元定義
- `zod` スキーマから TypeScript 型を自動生成
- API レスポンスを `parse` で検証


### C-12: 編集ページのモバイル 2 カラム崩れ

**担当観点**: Designer（主）\
**工数見積**: S\
**期待効果**: モバイルでの編集操作性向上。

**改善内容**:

- モバイル時にファイル一覧をアコーディオンまたは Drawer に変更
- カード配置エリアをメイン表示にし、ファイル追加は FAB ボタンから


### C-13: ランドマーク・スキップリンクの不完全さ

**担当観点**: A11y（主）\
**工数見積**: XS\
**期待効果**: スクリーンリーダーでのページ構造把握が容易に。

**改善内容**:

- `layout.tsx` の `<main id="main-content">` の位置を確認・修正
- ヘッダーに `role="banner"`、フッターに `role="contentinfo"` を明示
- ナビゲーション領域に `role="navigation"` と `aria-label` を付与


### C-14: API キャッシュ戦略の欠如

**担当観点**: Engineer（主）\
**工数見積**: M\
**期待効果**: S3 アクセス頻度の削減、レスポンス速度向上。

**改善内容**:

- GET `/api/sites/layout` に `Cache-Control: public, max-age=60, stale-while-revalidate=300` を追加
- GET `/api/docs` にも同様のキャッシュヘッダーを追加
- PUT 時に `revalidatePath` でキャッシュ無効化


### C-15: ドキュメント閲覧ページの戻る導線・リトライ欠如

**担当観点**: Designer（主）、A11y（副）\
**工数見積**: XS\
**期待効果**: エラー時のユーザー離脱防止。

**改善内容**:

- エラーアラート内に「再読み込み」ボタンを追加
- ボタンクリックで `fetchContent()` を再実行

> **【A11y 推奨】** 再読み込みボタンに `aria-label="ドキュメントを再読み込み"` を付与。


### C-16: 動的ページのメタデータ・サイトマップ欠如

**担当観点**: Engineer（主）\
**工数見積**: M\
**期待効果**: SEO 効果の向上、SNS シェア時のプレビュー改善。

**改善内容**:

- `docs/view/page.tsx` に `generateMetadata` を追加（ドキュメントタイトルを動的取得）
- `sitemap.ts` にレイアウト JSON のカード一覧を動的追加


### C-17: フォーカスインジケーター・リンク視認性

**担当観点**: A11y（主）、Designer（副）\
**工数見積**: S\
**期待効果**: キーボードナビゲーション時の操作位置が明確に。

**改善内容**:

- グローバル CSS に `focus-visible` スタイルを追加（3px solid アウトライン）
- リンクに `text-decoration: underline` を常時適用（ホバー時のみでなく）

> **【Designer・A11y 対立点】** リンクのアンダーライン常時表示について。
>
> Designer: デザインの洗練さが低下する。
>
> A11y: WCAG 1.4.1 では色以外の手段でリンクを識別可能にする必要がある。
>
> **推奨案**: ナビゲーション内リンクはアンダーラインなし（周囲のコンテキストで識別可能）、本文中のリンクはアンダーライン常時表示。


### C-18: `EditBody.tsx` の巨大化

**担当観点**: Engineer（主）\
**工数見積**: M\
**期待効果**: 保守性・テスタビリティの向上。

**改善内容**:

- `FileListPanel.tsx`: ファイル一覧・アップロード・削除を分離
- `CardAreaPanel.tsx`: D&D カード配置エリアを分離
- `useLayoutEditor.ts`: 状態管理をカスタムフックに抽出

| `zod` 導入によるバンドルサイズ増加 | 低 | サーバーサイドのみで使用し、クライアントバンドルに含めない |
| RSC 化による既存動作の regression | 中 | 段階的に移行し、各ステップで E2E テストを実施 |
| 認証方式変更時の管理者への影響 | 中 | 移行期間を設け、Basic 認証と新方式を並行運用 |
| `@dnd-kit` の KeyboardSensor がモバイル Safari で未検証 | 低 | 実機テストで確認。\
問題があれば上下ボタンによる代替 UI を追加 |
| ダークモードのコントラスト修正が他のスタイルに波及 | 低 | テーマの `contrastText` で一元管理し、個別 `sx` での色指定を削減 |

> **未解決課題**:

- フッターの情報設計（C-21）は IA（情報アーキテクチャ）の全体見直しが必要。\
サイト規模拡大時に再検討。
- タイポグラフィスケールの統一（C-22）は MUI テーマの `typography` で一括定義可能だが、`editor-core` との整合性確認が必要。
- Service Worker の PWA 対応（E-022）は Capacitor ビルドとの兼ね合いがあり、別プロジェクトとして扱う。
